# Тестирование Stasis

> **Статус: черновик.** Философия и слои адаптированы из `di-sukharev/vibe`
> (`docs/TESTING.md`) под стек Stasis: **Vitest + pnpm**, `node:sqlite`, Fastify `inject`,
> initData-auth, детерминированный engine с golden-тестами. Playwright у нас ещё НЕ
> настроен — раздел про e2e помечен как «to build».

## Философия

Пирамида тестов: покрываем на **самой высокой полезной поведенческой границе**, но не
дублируем. E2E — только когда риск виден пользователю и пересекает границу клиент↔сервер
(критичные сценарии, восстановление сессии, персистентность, важные пустые/ошибочные состояния).

Правила:

- Не добавлять e2e просто потому, что в коде есть ветка — добавлять, когда он предотвращает
  правдоподобную регрессию.
- Комбинаторные матрицы валидации и перебор ответов — **не** в e2e, а в unit/integration.
- Граничные случаи (валидация, отказ прав, протухшая сессия, пустые данные, дубли записи)
  живут в unit/integration-слое.
- **Engine детерминирован** — покрывается golden-тестами, не e2e. «Недетерминизм» допустим
  только в *формулировке* результата, никогда в выборе карточки (CLAUDE.md).

## Слои и что где тестировать

| Слой | Где в репо | Что проверяет |
|---|---|---|
| **Контракт** | `packages/shared/src/*.test.ts` | zod-схемы `SubmitPayload` и т.п. — единый источник клиент↔сервер |
| **Engine (golden)** | `apps/server/src/engine/*.test.ts` | чистые детерминированные функции: scoring, render, выбор карточки |
| **Крипто/поля** | `apps/server/src/crypto/field.test.ts` | шифрование ПД (152-ФЗ-контур) |
| **Auth** | `apps/server/src/auth/*.test.ts` | верификация initData, выпуск/проверка JWT-сессии |
| **Сервер (integration)** | `apps/server/src/app*.test.ts` | реальные роуты через Fastify `app.inject` + `node:sqlite` |
| **Content** | `apps/server/src/content/loader.test.ts` | загрузка/валидация YAML-матрицы, запрет forbidden-лексики |
| **Мини-апп (компоненты)** | `apps/miniapp/**/*.test.ts` | Vitest — UI-логика |
| **E2E мини-аппа** | *(to build)* | Playwright-смоук критичного пути (см. ниже) |

## Команды (реальные скрипты Stasis)

```bash
pnpm test          # pnpm -r test — все пакеты (корневой скрипт)
pnpm typecheck     # pnpm -r typecheck
pnpm build         # pnpm -r build

# отдельный пакет:
pnpm --filter @stasis/server test
pnpm --filter @stasis/shared test
pnpm --filter @stasis/miniapp test

# один файл / watch (внутри пакета):
pnpm --filter @stasis/server exec vitest run src/engine/scoring.test.ts
pnpm --filter @stasis/server exec vitest        # watch
```

## Интеграция сервера: паттерн

Тестируем через `buildApp(...)` + Fastify `app.inject` (в репо уже так — `app.test.ts`,
`app.submit.test.ts`). БД поднимаем на временном/in-memory `node:sqlite`, миграции
прогоняем перед тестом, никогда не трогаем прод-файл.

initData в тестах **подписываем тестовым `BOT_TOKEN`** тем же алгоритмом, что и
`verifyInitData` (`apps/server/src/auth/init-data.ts`) — так `POST /auth` проходит без
реального Telegram. Секреты сессии/шифрования — фиктивные тест-значения
(`JWT_SECRET`, `DATA_ENC_KEY` = валидный 64-hex).

Минимум, что должен покрывать серверный слой (частично уже есть):

- `POST /auth`: валидный initData → токен; протухший/подделанный → 401.
- `/me`, `/submit` без/с битым `Bearer` → 401; валидный токен, но удалённый user → 401.
- `POST /submit`: невалидный payload → 400; валидный → `{ profileId, result }`,
  профиль пересчитан **на сервере** (клиенту не верим — CLAUDE.md).
- Персистентность: сохранённый run читается обратно, ПД в БД зашифрованы.

## E2E мини-аппа (to build)

Когда настроим Playwright — смоук критичного пути в реальном браузере против реального
сервера и Vite-сборки:

- поднять тестовый сервер на изолированном порту с временной SQLite (имя файла должно
  оканчиваться на `_test`, чтобы исключить порчу боевых данных);
- засеять тестового пользователя без утечки секретов в браузер;
- **мокнуть Telegram initData**: подписать тест-payload тест-`BOT_TOKEN` и подставить как
  `window.Telegram.WebApp.initData` (мини-апп шлёт его на `/auth`);
- сценарий: согласия + 18+ → колесо → тест → результат (карточка + шаг) → сохранение читается
  обратно;
- проверить дисклеймер «не диагноз / не терапия» на входе и на результате;
- проверить безопасный блок с ресурсами поддержки при критичной нересурсности.

## CI (to build)

Прогонять на PR и в `master`: `pnpm typecheck` → `pnpm test` → `pnpm build`, затем (когда
появится) Playwright-смоук. Тестовая БД — только файл с суффиксом `_test`.

## Чего НЕ делать

- Не гонять комбинаторику ответов теста через e2e — это golden/unit в engine.
- Не проверять выбор карточки недетерминированно — engine обязан быть детерминированным.
- Не копировать валидированные шкалы (MBI/PHQ) в тест-фикстуры дословно (CLAUDE.md).
- Не давать клиенту вычислять профиль/слабую сферу в тест-обвязке — только сервер.
